Artifact Library

A working reference for Windows forensic artifacts. Each entry covers what the artifact is, how to triage it under time pressure, and a repeatable parsing workflow. Tooling references the Eric Zimmerman suite where applicable.

Execution

Prefetch (.pf)

▸

Definition

Windows performance-optimization files that record application launches. Each .pf file stores the executable name, run count, up to eight most recent run timestamps (Win8+), referenced files and directories, and volume information. Strong evidence of program execution.

C:\Windows\Prefetch\*.pf

Triage Tips

  • Sort by last run time first; recent execution of LOLBins (psexec, wmic, rundll32, certutil) from odd paths is a fast win.
  • An empty Prefetch folder on a workstation is itself suspicious — check if it was wiped or if the host is a server/SSD with prefetching disabled.
  • The hash in the filename encodes the executable's path — two .pf files for the same EXE name mean it ran from two locations.
  • Referenced-files list often exposes the payload an installer or loader touched.

Parsing Steps

  1. Collect the Prefetch directory from the triage image or live host (KAPE target: Prefetch).
  2. Parse with PECmd.exe -d C:\Cases\Prefetch --csv C:\Cases\out to process the whole directory.
  3. Win10/11 prefetch is MAM-compressed; PECmd handles decompression automatically — older tools may not.
  4. Review the timeline CSV for run timestamps; cross-reference run counts against ShimCache and Amcache.
  5. Pivot on the files-referenced output to identify dropped payloads and DLLs loaded at launch.
Execution

ShimCache (AppCompatCache)

▸

Definition

The Application Compatibility Cache, stored in the SYSTEM registry hive. Records full file paths and the file's $SI last-modified time for executables the OS has examined. On Windows 8+ an entry proves the file existed at that path — not that it executed.

SYSTEM\CurrentControlSet\Control\Session Manager\AppCompatCache

Triage Tips

  • Entries are only flushed to the registry on shutdown/reboot — a live hive may be missing the most recent activity. Pull volatile data or check hive transaction logs.
  • Report it as evidence of presence, not execution; corroborate with Prefetch or Amcache before claiming a binary ran.
  • Entry order is significant on some Windows versions — newer entries sit at the top.
  • Great for spotting renamed tools: the path survives even if the file was deleted.

Parsing Steps

  1. Acquire the SYSTEM hive plus its transaction logs (SYSTEM.LOG1, SYSTEM.LOG2).
  2. Replay dirty hives first — AppCompatCacheParser and registry tools handle this with the logs present.
  3. Run AppCompatCacheParser.exe -f SYSTEM --csv C:\Cases\out.
  4. Check entries in every ControlSet, not just the current one — older sets can hold deleted history.
  5. Stack the output with Amcache and Prefetch results to build an execution-evidence matrix per binary.
Execution

Amcache.hve

▸

Definition

A registry hive populated by application-compatibility processes. Its InventoryApplicationFile entries record full path, file size, version metadata, and a SHA-1 hash (of the first ~31MB) for executables observed on the system — including files that have since been deleted.

C:\Windows\AppCompat\Programs\Amcache.hve

Triage Tips

  • The free SHA-1 is the headline: pivot every unrecognized hash through VirusTotal or internal threat intel immediately.
  • Entry creation is event-driven (compatibility scans), so timestamps reflect inventory time, not first execution — be careful in reporting.
  • Hashes of large files are truncated at ~31MB of input; a mismatch with the file on disk isn't automatically tampering.
  • Look for unsigned binaries in user-writable paths (Downloads, Temp, ProgramData).

Parsing Steps

  1. Collect Amcache.hve and its .LOG transaction files from C:\Windows\AppCompat\Programs\.
  2. Parse with AmcacheParser.exe -f Amcache.hve -i --csv C:\Cases\out (-i includes program entries).
  3. Extract the SHA-1 column and batch-query your intel sources.
  4. Join results against ShimCache/Prefetch by path to upgrade "present" findings toward "executed."
  5. Review DriverBinary entries for suspicious or vulnerable drivers (BYOVD tradecraft).
Filesystem

$MFT (Master File Table)

▸

Definition

The core NTFS metadata structure: one record per file and directory on the volume, holding names, sizes, timestamps ($STANDARD_INFORMATION and $FILE_NAME), and data run locations. Small files may be resident — stored entirely inside the MFT record itself.

\\.\C:\$MFT (volume root, hidden system file)

Triage Tips

  • Compare $SI vs $FN timestamps: $SI is trivially timestomped via API, $FN generally is not. Divergence is a red flag.
  • Deleted-but-not-overwritten records persist; an MFT pull recovers names and metadata of removed tooling.
  • Resident data means tiny files (scripts, configs, beacons under ~700 bytes) can be recovered straight from the MFT.
  • Filter on creation times within the intrusion window and paths like Temp, ProgramData, Public, and PerfLogs.

Parsing Steps

  1. Acquire $MFT with a raw-capable collector (KAPE target: $MFT) — standard copy won't read it.
  2. Parse with MFTECmd.exe -f $MFT --csv C:\Cases\out.
  3. Generate a bodyfile or load the CSV into Timeline Explorer; filter to the incident window.
  4. Flag records where $FN creation predates $SI creation, or where $SI has zeroed sub-second precision (common timestomp tell).
  5. Cross-reference suspicious entries against $UsnJrnl to recover rename/delete history.
Filesystem

$UsnJrnl:$J (USN Change Journal)

▸

Definition

NTFS's change journal: an append-only log of file system operations (create, delete, rename, data overwrite, attribute change) with timestamps and reason codes. It records actions, not just final state — making it the volume's short-term memory.

\$Extend\$UsnJrnl (alternate data stream $J)

Triage Tips

  • Look for rapid create→rename→delete sequences — classic staging and cleanup behavior.
  • Journal records survive after the file itself is gone; this is often the only proof a tool ever existed.
  • Retention is typically days to weeks on a busy volume — collect it early before it rolls.
  • Mass FILE_DELETE + DATA_OVERWRITE reason codes in a tight window can mark ransomware detonation or wiper activity.

Parsing Steps

  1. Collect the $J stream (KAPE target: $J); it's sparse, so expect a large logical size with smaller real data.
  2. Parse with MFTECmd.exe -f $J --csv C:\Cases\out — supply -m $MFT as well to resolve full paths.
  3. Filter by reason codes of interest: FileCreate, FileDelete, RenameNewName.
  4. Reconstruct lifecycles for files of interest by following their MFT entry/sequence numbers through the journal.
  5. Merge with the MFT timeline to expose timestomping and deleted-file activity.
User Activity

ShellBags

▸

Definition

Registry structures that store Explorer folder-view preferences. As a side effect they record which folders a user account browsed interactively — including folders on removable media, network shares, and paths that have since been deleted.

USRCLASS.DAT → Local Settings\Software\Microsoft\Windows\Shell\BagMRU

Triage Tips

  • Per-user artifact: pull USRCLASS.DAT (and NTUSER.DAT) for every account of interest, including local admin accounts attackers favor.
  • Evidence of interactive, hands-on-keyboard browsing — strong signal when distinguishing human operators from automated malware.
  • Traversal into staging paths (e.g. C:\PerfLogs, C:\Users\Public) or to UNC paths of other hosts maps lateral interest.
  • MRU ordering and timestamps let you sequence which folders were opened most recently.

Parsing Steps

  1. Collect USRCLASS.DAT from %LocalAppData%\Microsoft\Windows\ plus transaction logs for each account.
  2. Parse with SBECmd.exe -d C:\Cases\hives --csv C:\Cases\out (or interactively in ShellBags Explorer).
  3. Note First Interacted / Last Interacted timestamps and absolute paths, flagging deleted or removable-media paths.
  4. Sequence folder access alongside logon sessions from event logs to attribute activity to a session.
  5. Diff ShellBag paths against the current filesystem to enumerate folders the attacker removed.
Filesystem

SRUM (SRUDB.dat)

▸

Definition

The System Resource Usage Monitor, an ESE database recording roughly 30–60 days of per-application resource accounting: network bytes sent/received per app per hour, process CPU time, and the user SID responsible. Effectively built-in flight-recorder telemetry.

C:\Windows\System32\sru\SRUDB.dat

Triage Tips

  • Use network-usage tables to bound data exfiltration: bytes-sent per application per hour is hard evidence when flow logs are missing.
  • A high-upload entry from an unexpected binary (or a renamed archiver) is a fast exfil lead.
  • Data is flushed from memory to the database roughly hourly — the last hour before shutdown may be absent.
  • Attribution: each entry ties to a user SID, helping pin activity to a compromised account.

Parsing Steps

  1. Collect SRUDB.dat plus the SOFTWARE hive (needed to resolve interface and profile IDs).
  2. If the ESE database is dirty, repair it first: esentutl /r sru /i then esentutl /p SRUDB.dat on a working copy.
  3. Parse with SrumECmd.exe -f SRUDB.dat -r SOFTWARE --csv C:\Cases\out.
  4. Sort NetworkUsage output by bytes sent within the incident window; convert totals into exfil estimates.
  5. Correlate spikes with Prefetch run times and firewall/proxy logs to identify the exfil channel.
User Activity

Windows Event Logs (.evtx)

▸

Definition

The native Windows logging system: binary XML records across hundreds of channels covering authentication, service installs, scheduled tasks, PowerShell activity, RDP sessions, and more. The backbone of most intrusion timelines.

C:\Windows\System32\winevt\Logs\*.evtx

Triage Tips

  • First-pass channels: Security, System, PowerShell/Operational, TaskScheduler, TerminalServices-LocalSessionManager, Sysmon if present.
  • High-value IDs: 4624/4625 (logons), 4672 (privileged logon), 7045 (service install), 4698 (scheduled task), 1102 (audit log cleared).
  • Event ID 1102 or large gaps in record numbers indicate log clearing — treat the gap itself as a finding and pivot to forwarded logs or VSS copies.
  • Type 3 logons with NTLM from workstation-to-workstation are a classic lateral movement signature.

Parsing Steps

  1. Collect the full winevt\Logs directory (KAPE target: EventLogs) — don't cherry-pick channels at collection time.
  2. Parse with EvtxECmd.exe -d C:\Cases\Logs --csv C:\Cases\out using the community maps for field normalization.
  3. Load into Timeline Explorer; filter to the incident window and the high-value Event IDs above.
  4. Build a logon-session table (4624 + LogonID) to anchor every other artifact to an authenticated session.
  5. Check volume shadow copies for .evtx versions predating any log clearing.
